Git入門：一人でもチームでも使えるバージョン管理の基本

Gitは、プログラムのソースコードなどの変更履歴を記録・追跡するためのバージョン管理システムである。一人での開発効率を上げるだけでなく、チームでの共同作業に不可欠なツールだ。この記事では、Gitの基本的な使い方から、GitHubと連携した実践的な開発フローまでを解説する。

第1章: Gitの初期設定と基本の3ステップ

最初にPCでGitを使うための準備を行う。この設定は一度だけ実施すればよい。

1-1. ユーザー情報の設定

コミット履歴に「誰が変更したか」を記録するため、名前とメールアドレスを設定する。

# グローバル設定（このPC上のすべてのGitリポジトリで使われる）
git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"

【補足】グローバル設定とローカル設定

--global はPC全体の設定。特定のプロジェクト（フォルダ）だけ違う名前を使いたい場合、そのフォルダで --global なしでコマンドを実行する。ローカル設定が優先される。

1-2. リポジトリの作成

バージョン管理を始めたいフォルダに移動し、次のコマンドを実行する。

# "my-project" フォルダに移動
cd my-project

# このフォルダをGitリポジトリとして初期化する
git init

これを実行すると、フォルダ内に .git という隠しフォルダが作成される。ここに変更履歴などの全情報が保存される。

1-3. 基本の3ステップ：「作業 → add → commit」

Gitでの作業は、基本的に以下の3ステップの繰り返しになる。

  1. 作業: ファイルを作成・編集する。
  2. ステージング (git add): 変更したファイルの中から、履歴に記録したいものを選ぶ。
  3. コミット (git commit): 選んだファイルの変更内容を、メッセージ付きで履歴に記録する。
# 現在のディレクトリ以下のすべての変更をステージングエリアに追加
git add .

# ステージングされた内容を "最初のコミット" というメッセージ付きで履歴に記録
git commit -m "最初のコミット"

【用語解説】

  • ワーキングディレクトリ: 現在作業しているファイルが存在するフォルダ。
  • ステージングエリア（インデックス）: コミット対象の変更を一時的に置く場所。「次のコミットに含める変更リスト」のようなもの。
  • リポジトリ: .git フォルダのこと。ファイルの全変更履歴が保存されている場所。

第2章: 履歴の確認と過去の参照

コミットを繰り返すことで、プロジェクトの歴史が作られていく。

2-1. コミット履歴の確認

# これまでのコミット履歴を一覧で表示
git log

# 履歴を1行で、ブランチの関係もグラフィカルに、すべてのブランチを表示（推奨）
git log --oneline --graph --all

2-2. 変更内容（差分）の確認

# 前回のコミットから、現在のワーキングディレクトリの変更内容を確認
git diff

2-3. 過去のバージョンを参照

特定の過去の状態に戻り、ファイルを確認することができる。

# <commit_id> には git log で表示されるコミットID（例: a1b2c3d）を指定
git checkout <commit_id>

注意: この状態でファイルを編集してコミットすると歴史が分岐し複雑になる。ファイル確認後は、元の最新の状態に戻るのが一般的。

# 最新の状態に戻る
git checkout main

第3章: ブランチを使った効率的な開発

ブランチはGitの強力な機能。安全かつ並行して作業を進めるために活用する。

3-1. ブランチとは？

履歴の流れを分岐させ、元の本流（mainブランチ）に影響を与えずに作業を進めるための機能。「料理のレシピ（main）のコピーを作り、新しい味付け（featureブランチ）を試作する」ようなもの。試作が成功すれば、元のレシピに反映（マージ）する。

3-2. 個人開発でもブランチは役立つか？

YES。非常に役立つ。

  1. 安全な作業場所の確保（試行錯誤がしやすい）

    mainブランチを常に「正常に動作する安定版」とし、新機能追加やバグ修正は別ブランチで行う。作業が混乱しても、そのブランチを捨ててmainからやり直せるため、安心して試行錯誤できる。

  2. 複数のタスクを並行して進められる

    「機能Aを実装中に、バグBを見つけた」という状況でも、ブランチがあれば対応は容易。

    1. feature/A ブランチの作業を一旦コミット。
    2. main ブランチに戻り、fix/B ブランチを作成してバグを修正し、main にマージ。
    3. 再び feature/A ブランチに戻り作業を再開。

    このように、思考をクリアに保ったままタスクを切り替えられる。

3-3. ブランチの基本操作

# ブランチの一覧を表示
git branch

# "feature/new-login" という名前のブランチを作成
git branch feature/new-login

# 作成したブランチに切り替え（移動）
git checkout feature/new-login

# ブランチの作成と切り替えを同時に行う
git checkout -b feature/new-login

# 最近のGitでは、より直感的な switch コマンドが推奨されている
git switch feature/new-login      # ブランチの切り替え
git switch -c feature/new-login   # ブランチの作成と切り替え

# 不要になったブランチを削除（マージ済みのもののみ）
git branch -d feature/new-login

第4章: ブランチの統合（マージ）

分岐したブランチでの作業完了後、その成果を main ブランチに取り込む。この操作が「マージ」である。

4-1. マージの基本

# 1. mainブランチに移動する
git switch main

# 2. "feature/new-login" ブランチの変更を取り込む
git merge feature/new-login

4-2. マージの種類

マージには、状況に応じて2つの種類がある。

  • Fast-forward マージ（早送りマージ）

    mainブランチに変更がなく、作業ブランチだけが進んでいる場合、mainのポインタが作業ブランチの先端に移動するだけ。履歴は一本線のままになる。

  • 非Fast-forward マージ（マージコミットを作成）

    mainと作業ブランチの両方で変更が進んでいる場合、2つの分岐した履歴を統合するための新しい「マージコミット」が自動的に作成される。これにより、「ここで機能が取り込まれた」という履歴が明確に残る。

コマンド Fast-forward可能な場合 Fast-forward不可能な場合 主な用途
git merge <branch> (デフォルト) Fast-forwardマージを行う マージコミットを作成する 個人開発、シンプルな運用
git merge --no-ff <branch> マージコミットを強制的に作成する マージコミットを作成する 機能開発の履歴を残したいチーム開発
git merge --ff-only <branch> Fast-forwardマージを行う エラーを出して停止する リモートの変更を取り込むなど、安全性を重視する場合

4-3. コンフリクト（衝突）の解決

異なるブランチで同じファイルの同じ行を編集すると、Gitはどちらを優先すべきか判断できず、「コンフリクト（衝突）」が発生する。

  1. コンフリクトが起きたファイルをエディタで開く。
  2. <<<<<<<, =======, >>>>>>> のマーカーで示された箇所を、手動で正しい内容に修正する。
  3. 修正後、ファイルを保存し、git add と git commit を行いマージを完了させる。

マージを中止して元の状態に戻すことも可能。

# マージを中止して、マージ開始前の状態に戻る
git merge --abort

第5章: 作業の取り消しと修正

作業を元に戻したい場合の方法。

5-1. ファイルの変更を取り消す (git restore)

# ワーキングディレクトリでの変更を取り消す（前回のコミットの状態に戻す）
git restore <file_name>

# ステージングした変更を取り消す（ワーキングディレクトリの変更は残る）
git restore --staged <file_name>

5-2. コミットをやり直す

直前のコミットを修正する (--amend)

コミットメッセージの間違いや、ファイルの入れ忘れ時に使用する。

# ファイルを追加し忘れた場合
git add forgotten_file.txt

# 直前のコミットを修正（新しいコミットは作られず、直前のコミットが上書きされる）
git commit --amend

過去のコミットを取り消す (reset vs revert)

  • git reset <commit_id>: 指定したコミットまで歴史を巻き戻す。push済みのコミットをresetすると、他者の履歴と矛盾が生じるため危険。 個人作業でのみ使用する。
  • git revert <commit_id>: 指定したコミットの変更内容を打ち消す、新しいコミットを作成する。 歴史を書き換えないため、チームで共有しているリポジトリでも安全に使える。

第6章: GitHubとの連携

ローカルリポジトリをGitHubなどのリモートサーバーに保存・公開することで、バックアップやチームでの共同作業が可能になる。

6-1. リモートリポジトリの接続とPush（アップロード）

  1. GitHubでリポジトリを作成: GitHubサイトで新規リポジトリを作成。

    （例）リポジトリ名：yourrepo

  2. リモートリポジトリを登録: ローカルリポジトリに、接続先となるGitHubリポジトリのURLを登録する。 
    git remote add origin https://github.com/yourname/yourrepo.git

#リモートリポジトリが正しく登録されたか確認
git remote -v

# 実行結果例
# origin  https://github.com/yourname/yourrepo.git (fetch)
# origin  https://github.com/yourname/yourrepo.git (push)
  3. ローカルの変更をPush: ローカルのコミット履歴をGitHubにアップロード（Push）する。 
    # デフォルトのブランチ名を "main" に設定（初回のみ）
git branch -M main

# mainブランチの内容を、origin（GitHub）にPushする
# -u オプションは、今後のpushのデフォルト先として設定
git push -u origin main

VSCodeの活用

VSCodeの「ソース管理」パネルを使えば、add, commit, push, pull などの操作をボタンクリックで行える。「Publish to GitHub」ボタンは、リポジトリ作成から最初のPushまでを自動化する。

6-2. GitHubへの認証

git push を実行すると、GitHubから認証を求められる。これは、権限のない第三者がリポジトリを勝手に書き換えるのを防ぐためである。認証方法は主に2つある。

認証方法 主な用途 特徴
SSHキー (推奨) 個人の開発マシンでの日常作業 一度設定すればパスワード入力不要。安全性が高い。
Personal Access Token (PAT) 自動化スクリプト (CI/CD)、API利用 有効期限や権限を細かく設定可能。セキュリティリスクを限定できる。

日常的な開発では、設定の手間は最初だけなのでSSHキーが最もストレスが少ない。

6-3. 認証方法①: Personal Access Token (PAT)

git push 時に表示される認証ダイアログでは、パスワードの代わりにPATを使用する。

  • ユーザ名: GitHubのユーザ名
  • パスワード: 作成したPATを貼り付け

PATの作成手順:

  1. GitHub右上のプロフィール画像 → Settings
  2. 左メニュー最下部の Developer settings
  3. Personal access tokens → Fine-grained tokens を選択
  4. Generate new token をクリック
  5. Token name（例: my-mac-token）、Expiration（有効期限）を設定
  6. Repository access でリポジトリを選択
  7. Permissions → Repository permissions → Contents を探し、Access: Read and write に変更
  8. Generate token をクリックして生成。このトークンは一度しか表示されないので、必ずコピーして安全な場所に保管する。

6-4. 認証方法②: SSHキー (推奨)

SSHキーを設定すると、git push 時に毎回パスワードやPATを入力する必要がなくなる。設定方法は公開鍵と秘密鍵のペアを作成し、公開鍵をGitHubに登録する。

詳細な手順は以下の記事などを参照するといい。

6-5. Push時の認証エラーと対処法

エラーメッセージ:

remote: Invalid username or token. Password authentication is not supported for Git operations.

原因:

現在サポートされていない古いパスワード認証の情報が、PC内に保存されている場合にこのエラーが出る。

対処法:

古い認証情報をクリアする必要がある。

  1. Git設定から認証ヘルパーを削除 (効果がない場合もある) 
    git config --global --unset credential.helper
  2. OSの資格情報マネージャからGitHubの認証情報を削除 (Windowsの場合)
    1. スタートメニュー → 「資格情報マネージャー」 (Credential Manager) を検索して開く。
    2. 「Windows 資格情報」を選択。
    3. 一覧から git:https://github.com や github.com に関連するエントリを探して削除する。
    4. 再度 git push を実行すると、新しい認証ダイアログが表示されるはず。

(macOSの場合は「キーチェーンアクセス」アプリで同様の操作を行う)

第7章: やっておくと便利な設定

.gitignore ファイル

Gitの管理対象から外すファイルを指定する設定ファイル。プロジェクトのルートディレクトリに .gitignore という名前で作成する。

なぜ必要か？

パスワードなどの機密情報や、OSが自動生成するファイル、ビルド時に生成されるファイルなどをリポジトリに含めないようにするため。

記述例 (.gitignore)

# 一時ファイル
*.log

# OSのファイル
.DS_Store
Thumbs.db

# ビルド結果
/dist/
*.class

# 環境設定ファイル（機密情報）
.env

不足している知識と、次へのステップ

ここまでの内容で、Gitの基本は網羅できている。さらにGitを使いこなすために、以下のトピックについて調べると良い。

  1. git rebase: merge と並ぶもう一つの統合方法。コミット履歴を一直線に綺麗に保つことができるが、概念が少し複雑。マージとの違いから理解を始めるのがよい。
  2. git stash: コミットしたくない作業内容を一時的に退避させる機能。ブランチを切り替える前に、中途半端な作業を隠したい時に便利。
  3. Pull Request (プルリクエスト): チーム開発の要。自分の変更をmainブランチにマージしてもらう前に、「この変更をレビューしてください」と依頼する機能。GitHub上で行う操作であり、コードの品質維持に重要。
  4. Git-flowなどのブランチ戦略: チーム開発を円滑に進めるための、ブランチの命名規則や運用ルールのモデル。「どのような時に、どのような名前のブランチを切り、どうやってマージしていくか」という一連の流れを定義したもの。
