業界・業種・業務形態・職種の定義

業界：どの"世界"にいるか（例：IT、金融、製造）。

業種：会社が何で収益を上げているか（例：SIer、Webサービス、SES）。

業務形態：仕事の進め方・契約の形（例：受託開発、準委任、SES、社内開発）。

職種：個人の役割・担当業務（例：開発エンジニア、SE、PM、SRE）。

業種（会社タイプ）の深掘り

SIer（システムインテグレーター）

  • ビジネスモデル：企業の業務システムを受託・構築・運用する。要件定義〜保守まで担うことが多い。
  • 主な収入源：案件受注（請負）／保守契約／派遣（SES）
  • メリット：業務知識が身につく、大規模案件経験が得られる。安定性が高い。
  • デメリット：プロジェクト単位で動くためリリース頻度が低い、常に要件変更が起きやすい。

Webサービス（自社プロダクト／SaaS）

  • ビジネスモデル：自社のサービス（BtoC/BtoB）を開発・運用して継続課金を得る。
  • 主な収入源：サブスク／広告／取引手数料
  • メリット：スピード感のある開発、ユーザーデータに基づく改善が可能。
  • デメリット：事業リスク（売上が伸びなければ厳しい）、少数精鋭の現場が多い。

パッケージベンダー

  • ビジネスモデル：業務パッケージ製品を開発しライセンス販売／保守を行う。
  • 主な収入源：製品販売、ライセンス、保守費
  • 特徴：導入支援でSIerと絡むことが多い。製品中心の開発。

SES（システムエンジニアリングサービス）事業者

  • ビジネスモデル：顧客に技術者（人）を派遣・常駐させることで収益を上げる。プロジェクト単位で技術者を提供。
  • 主な収入源：常駐契約（月額の人件費マージン）
  • メリット：人材が中心のため採用ができれば安定収益を作りやすい。
  • デメリット：技術者は常駐先に左右されるため、経験の幅が限られる場合がある。働き方（常駐）が合うかが重要。

SIer vs SES（よくある混同ポイント）

観点 SIer SES事業者
主な売り物 システム・ソリューション 技術者（リソース）
契約形態 案件ごとの請負や保守 常駐・期間契約（準委任が多い）
技術成長 案件を通じた幅広い経験が得られる 常駐先に依存、幅はまちまち

業務形態（契約・進め方）の深掘り

受託（請負）

成果物（システム）を納品する契約。成果物に対して責任が発生します。要件が固まっている案件に向きます。

準委任 / 業務委託

一定の工数・期間で作業を提供する（成果物保証は弱い）。仕様変更が多い案件や常駐サポートに向きます。

SES（客先常駐）

技術者を客先に常駐させて作業を行う。契約的には準委任が多く、業務範囲や評価が常駐先に依存します。

自社開発

自社で企画・開発・運用を行い、製品/サービスで収益を作ります。

オフショア / オンショア分業

実装を海外/別拠点に委託する。コストや工数の最適化に利用されます。

法的／契約上の違い（簡単に）

請負：成果物に対する責任がある。準委任：時間・体制の提供が中心。契約内容は雇用条件や責任範囲を明確にすること。

職種（役割別ガイド）

ここでは「入りやすいもの」を上に、下に行くほど専門性・難易度が上がる順で並べています。

Stage 1（入りやすい・未経験歓迎）

運用保守（監視・オペレーター）

  • 仕事内容：監視ダッシュボードの確認、アラート対応の一次切り分け、定常作業の実施
  • 必要スキル：基本的なOS操作、監視ツールの操作、ログの読み方、簡単なスクリプト
  • 採用ハードル：低い（未経験での採用が多い）
  • キャリアパス：運用 → インフラ構築補助 → SRE / インフラエンジニア

テスター（QA）

  • 仕事内容：テストケース作成・手動テスト、バグレポート作成、回帰テスト実行
  • 必要スキル：テスト設計の基礎、手順書作成能力、論理的思考
  • 採用ハードル：低い（未経験歓迎のポジション多数）
  • キャリアパス：QA → テスト自動化エンジニア → 品質保証リード

ヘルプデスク・サポート

  • 仕事内容：ユーザーからの問合せ対応、障害受付・一次対応、FAQの作成
  • 必要スキル：コミュニケーション、ログの読み取り、基本的なトラブルシューティング
  • 採用ハードル：低い（カスタマーサービス経験が活きる）
  • キャリアパス：サポート → 運用保守 / プロダクトサポートリード

簡単なプログラミング／Web系下流工程（補助）

  • 仕事内容：既存コードの修正、単体テスト記述、HTML/CSS の修正など補助作業
  • 必要スキル：基本的なプログラミング知識（HTML/CSS/簡単なJavaScriptやPHP等）、Git の基礎
  • 採用ハードル：低い（学習意欲と簡単な成果物があれば採用されやすい）
  • キャリアパス：下流開発 → アプリ開発エンジニア

Stage 2（中級：実務経験で飛躍できる）

インフラ構築補助

  • 仕事内容：環境準備、サーバ設定の補助、構成管理スクリプトの作成補助
  • 必要スキル：Linux基礎、シェルスクリプト、クラウドの入門知識
  • 採用ハードル：中（運用経験があると有利）
  • キャリアパス：インフラ構築 → クラウドアーキテクト / SRE

SE（下流：設計書通りに作る）

  • 仕事内容：詳細設計に基づく実装、データマッピング、結合テストの実務
  • 必要スキル：設計書の読み書き、基本的なDB操作、テストの理解
  • 採用ハードル：中（開発経験や下流工程経験があると有利）
  • キャリアパス：上流SE → PM / テクニカルリード

アプリ開発エンジニア（自分で設計してコードを書く）

  • 仕事内容：仕様に基づいた設計・実装・テスト・リファクタリング
  • 必要スキル：プログラミング言語の習熟、フレームワーク利用、DB設計の基礎
  • 採用ハードル：中（ポートフォリオや実務での小さな経験が重要）
  • キャリアパス：テックリード → アーキテクト / EM

インフラ構築（サーバ／ネットワーク本番構築）

  • 仕事内容：本番環境の設計・導入・冗長化・性能チューニング
  • 必要スキル：ネットワーク設計、負荷分散、構成管理（Terraform/Ansible）
  • 採用ハードル：中〜やや低（現場での作業経験が重要）
  • キャリアパス：クラウドアーキテクト / SRE / インフラマネージャー

Stage 3（上級：専門性・設計能力が求められる）

SE（上流：要件定義・基本設計）

  • 仕事内容：顧客ヒアリング、要件定義、基本設計、見積もり作成
  • 必要スキル：業務理解、設計ドキュメント作成、顧客折衝力
  • 採用ハードル：高め（実務経験・コミュニケーション力が必要）
  • キャリアパス：PM / コンサル / プロダクト責任者

SRE / DevOps

  • 仕事内容：信頼性設計、自動化、可観測性の実装、障害対応の仕組み作り
  • 必要スキル：プログラミング、クラウド運用、モニタリング設計（Prometheus等）
  • 採用ハードル：高め（運用と開発の両方の経験が必要）
  • キャリアパス：プラットフォーム責任者 / 信頼性アーキテクト

セキュリティエンジニア

  • 仕事内容：脆弱性診断、セキュリティ設計・運用、インシデント対応
  • 必要スキル：脆弱性知識、ログ解析、フォレンジックの基礎
  • 採用ハードル：高め（専門知識が求められる）
  • キャリアパス：CSIRTリード / セキュリティアーキテクト / コンサル

クラウドアーキテクト

  • 仕事内容：クラウドベースのシステム設計、コスト最適化、運用戦略の策定
  • 必要スキル：クラウド全体設計（AWS/GCP/Azure）、IaC、セキュリティ設計
  • 採用ハードル：高（豊富な現場経験が前提）
  • キャリアパス：CTO 付近の技術リード / クラウドコンサルタント

キャリアパターン（A/B/C）

  • パターンA（インフラの王道）：運用保守 → インフラ構築 → クラウド → SRE / クラウドアーキテクト
  • パターンB（ものづくり重視）：テスター／下流開発 → アプリ開発 → フロント/バックの専門 → テックリード
  • パターンC（専門性を極める）：運用 → セキュリティ基礎 → SOC/CSIRT → セキュリティエンジニア/コンサル

上記のキャリアパターンはあくまで代表的な一例です。個人の興味や適性、技術の進化によって、これら以外にも多様なキャリアパスが存在します。

未経験者向けロードマップ（実務で使える最短ルート）

  1. 選ぶ：まず興味のある職種を1つ決める（開発 or インフラ or 運用等）
  2. 基礎学習：言語基礎（開発）／Linux・ネットワーク基礎（インフラ）／テスト基礎（QA）
  3. 実践ポートフォリオ：小さな成果物を作る（個人アプリ、インフラ構築ノート）
  4. 応募：運用保守やQAなど入り口求人で実務経験を得る
  5. 専門化：現場で1〜2年の実務を積み、次の職種へ横移動または昇進

FAQ（よくある混同）

Q：SESは業種？業務形態？
A：両方で使われる。ページでは「SES事業者（業種）」と「客先常駐（業務形態）」を区別して説明しています。
Q：請負と準委任の違いは？
A：簡単に言うと、請負は成果物の納品責任がある。準委任は時間や体制の提供（成果保証は弱い）。
Q：自社開発と受託はどう見分ける？
A：自社開発は自社プロダクトを持ち収益化する会社。受託は他社の依頼で開発する会社。

用語集（短）

  • オンプレ：自社運用の物理サーバー
  • IaC：Infrastructure as Code（例：Terraform）
  • CI/CD：継続的インテグレーション／継続的デプロイ
  • SLO/SLI：サービス信頼性の指標
